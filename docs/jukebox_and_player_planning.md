# MusaX Jukebox & Standalone Player Design Plan

This document outlines the planning, design, and architecture for refactoring the Graphical Jukebox and building a compiled song player for MSX-DOS / MSX-DOS2.

---

## 1. Dynamic Song and FX Ingestion

### Current Issue
Currently, songs and sound effects are manually hardcoded in `src/jukebox/SRC/JUKEBOX.Z8A` and `src/jukebox/Makefile`. The header references (such as `SHALOMCASTLE.HEADER`) are hardcoded in the keyboard scanner `POLLKEY` (`BIT 1,D -> LD HL,SHALOMCASTLE.HEADER`), and the UI display strings are hardcoded in `PLAYLSTB` and `SNGNAMTB`. This requires editing the assembly codebase every time a song is added, modified, or removed.

### Proposed Solution: Autogenerated Playlist Integration
We will decouple the Jukebox assembly source code from the specific list of songs/FXs. The build system will scan a directory of songs and automatically generate a data definition file `TMP/PLAYLIST.Z8A`.

1. **Build Pipeline Automation**:
   - A Python helper script (e.g., `tools/generate_playlist.py`) will run before compilation.
   - It will scan the `SRC/TESTS/` folder for `.msl` files.
   - It will trigger `musax.py` to compile each `.msl` file to `TMP/*.Z8A`.
   - It will generate a consolidated `TMP/PLAYLIST.Z8A` assembly file containing all the necessary includes and metadata tables.

2. **Generated Data Structure (`TMP/PLAYLIST.Z8A`)**:
   The autogenerated file will contain:
   - Compile-time inclusions of all songs.
   - A central song table (`SONGTBL`) containing pointers to header locations, short names, and formatting strings.
   - A central FX table (`FXTBL`) structured similarly.
   - Constants defining the number of songs and FXs.

```assembly
; ==========================================
; AUTOGENERATED PLAYLIST - DO NOT EDIT MANUALLY
; ==========================================

; --- SONG BINARY INCLUSIONS
                INCLUDE "TMP/shalomcastle.Z8A"
                INCLUDE "TMP/adsr_test.Z8A"
                INCLUDE "TMP/lfo_test.Z8A"
                INCLUDE "TMP/1ch_chorus_test.Z8A"
                INCLUDE "TMP/sequencer_test.Z8A"
                INCLUDE "TMP/modulation_test.Z8A"

; --- SONG COUNT CONSTANT
SONGCOUNT       EQU     6

; --- DYNAMIC SONG TABLE
; Format: [HEADER_PTR (2b), NAME_CYAN_PTR (2b), NAME_WHITE_PTR (2b), SHORT_NAME_PTR (2b)]
SONGTBL:
                DEFW    SHALOMCASTLE.HEADER, SNG1_CYAN, SNG1_WHITE, SNG1_SHORT
                DEFW    ADSRTEST.HEADER,     SNG2_CYAN, SNG2_WHITE, SNG2_SHORT
                DEFW    LFOTEST.HEADER,      SNG3_CYAN, SNG3_WHITE, SNG3_SHORT
                DEFW    SEQTEST.HEADER,      SNG4_CYAN, SNG4_WHITE, SNG4_SHORT
                DEFW    MODTEST.HEADER,      SNG5_CYAN, SNG5_WHITE, SNG5_SHORT
                DEFW    SFXTEST.HEADER,      SNG6_CYAN, SNG6_WHITE, SNG6_SHORT

; --- STRINGS DEFINITION
SNG1_CYAN:      DEFB    #A1, ".shalom castle", 0
SNG1_WHITE:     DEFB    #A1, ".SHALOM CASTLE", 0
SNG1_SHORT:     DEFB    "SHALOM", 0

SNG2_CYAN:      DEFB    #A2, ".adsr test    ", 0
SNG2_WHITE:     DEFB    #A2, ".ADSR TEST    ", 0
SNG2_SHORT:     DEFB    "ADSR  ", 0
; ... (and so on)
```

In `JUKEBOX.Z8A`, we will replace the hardcoded tables and includes with:
```assembly
; --- INCLUDES
                INCLUDE "TMP/PLAYLIST.Z8A"
```
The assembler will resolve all symbols dynamically, keeping the code clean and automatically updated.

## 2. Interactive Selector Redesign

### Current Issue
The Jukebox uses hardcoded numeric keys (`1-6` for songs, `7-0` for FX) to select and trigger playback. This limits the playlist to 6 songs and 4 FXs and wastes UI space on key explanations.

### Proposed Solution: Two-Panel Navigation Grid
We will implement a standard file-manager layout with two side-by-side columns at the top of the Screen 1 grid:
- **Left Column**: Songs (Music tracks).
- **Right Column**: Sound Effects (FX tracks).

#### Redesigned Screen 1 Layout (32x24 Row-by-Row Specification)
```text
00:   MUSAX DYNAMIC JUKEBOX v2.0   
01: ────────────────────────────────
02: SNG:SHALOM   BPM:120 L:00 ST:PLAY  
03: ┌── SONGS ──────┐┌── SOUND FX ─┐
04: │▶1.SHALOM CAS ││ 1.FX_LO     │
05: │  2.ADSR TEST ││ 2.FX_HI     │
06: │  3.LFO TEST  ││ 3.FX_FAN    │
07: │  4.SEQ TEST  ││ 4.FX_TRI    │
08: │  5.MOD TEST  ││ 5.FX_JUMP   │
09: │  6.1CH CHOR  ││ 6.FX_EXPL   │
10: └───────────────┘└─────────────┘
11: ────────────────────────────────
12: CH TY VOL   NOTE INS  PER  STAT
13: 0A MU ████  A#4   02  #1FA PLAY
14: 1B MU ███▏  F-3   02  #2AC PLAY
15: 2C MU █     C-2   01  #5C2 PLAY
16: 3A FX ░░░░  ---   --  #000 OFF 
17: 4B FX ░░░░  ---   --  #000 OFF 
18: 5C FX ░░░░  ---   --  #000 OFF 
19: ────────────────────────────────
20:  KEYS: Cursors Nav | SPACE Play 
21:  S:Stop  P:Pause F:FadeO I:FadeI
22:  MUTE MUS: T:ChA  Y:ChB  U:ChC  
23:  MUTE FX : G:ChA  H:ChB  J:ChC  
```

#### Row Breakdown and Dimensioning:
- **Rows 00-01**: Centered program title header and static horizontal divider line.
- **Row 02**: Compact **Playback Info** line.
  - `SNG:` [6 chars for name, padded/truncated] (10 chars total)
  - `BPM:` [3 digits] (7 chars total)
  - `L:` [2 digits loop iter count] (4 chars total)
  - `ST:` [PLAY/PAUS/STOP/FADE] (8 chars total)
  - Layout: `SNG:SHALOM   BPM:120 L:00 ST:PLAY  ` (exactly 32 characters).
- **Rows 03-10**: Interactive **Two-Panel List Manager** (8 rows tall).
  - Left panel: `SONGS` (17 chars wide: cols 0 to 16). Lists 6 songs with a selection cursor (`▶` at col 1) and active-playing highlight casing.
  - Right panel: `SOUND FX` (15 chars wide: cols 17 to 31). Lists 6 FXs.
  - Selection cursors (`▶`) only draw on the currently active focused panel.
- **Rows 11-12**: Horizontal divider line and channel mixer column headers.
- **Rows 13-18**: Real-time **Channel Mixing Grid** (6 rows, 1 for each PSG audio stream).
  - Rendered using the existing caching optimizations (`CHCACHE`, volume visualizers `███`/`░░░`, note conversion table, detune frequency extraction, and ghost state monitoring).
- **Row 19**: Horizontal divider line separating mixing grid from controls.
- **Rows 20-23**: Compact **Teclas de Control / Keyboard Legend** (4 rows).
  - Row 20: General navigation rules (`Cursors` and `SPACE`).
  - Row 21: Playback modifiers (`S` for Stop, `P` for Pause, `F` for Fade Out, `I` for Fade In).
  - Row 22: Music channel mute controls (`T` for ChA, `Y` for ChB, `U` for ChC).
  - Row 23: Sound FX channel mute controls (`G` for ChA, `H` for ChB, `J` for ChC).


### Keyboard Navigation & Control Logic
We will read the Cursor keys and Action keys directly from the keyboard matrix mirrors in Page 3 RAM:
- **Row 8, Bit 4 (`UP`) / Bit 5 (`DOWN`)**: Moves the navigation cursor `▶` up and down in the currently focused panel.
- **Row 8, Bit 6 (`LEFT`) / Bit 7 (`RIGHT`)**: Toggles the active panel focus between the `SONGS` panel (left) and `SOUND FX` panel (right).
- **Row 8, Bit 0 (`SPACE`) or Row 7, Bit 7 (`RETURN`)**: Triggers the selected item in the focused panel.
  - If focused on `SONGS`: Stop current playback, load selected song header, and call `MUSPLAY`.
  - If focused on `SOUND FX`: Trigger the selected FX index via `FXPLAY`.

### Scrolling & Paging Algorithm
To support playlists larger than 6 items, we will implement dynamic windowing (scrolling):
- Keep two navigation index variables in RAM:
  - `CUR_PANEL`: Active focus (0 = Songs, 1 = FX).
  - `SEL_INDEX`: Selected item index (0 to `COUNT - 1`).
  - `SCR_START`: Scroll offset (0 to `COUNT - 6`). This specifies which song/FX starts rendering at the top of the list.
- When `SEL_INDEX` goes below `SCR_START`, set `SCR_START = SEL_INDEX`.
- When `SEL_INDEX` is greater than or equal to `SCR_START + 6`, set `SCR_START = SEL_INDEX - 5`.
- The drawing routine `DRAWPL` will iterate 6 times, rendering items from index `SCR_START` to `SCR_START + 5`.
- A scrollbar or page indicator (e.g., `Page X/Y` or small arrows `▲`/`▼`) will show the presence of off-screen items.

## 3. MSX-DOS Compiled Song Player & File Format

### The Assembly Address Problem
MusaX songs compiled to `.Z8A` contain absolute 16-bit pointers (e.g., jump addresses, phrase pointers, instrument maps). When assembled, they are targeted to a fixed memory address (`ORG`). In a typical MSX-DOS environment, a utility load address is unpredictable or must be flexible.

To solve this, we define a custom file format for compiled songs and explore three execution architectures.

### File Format Specification: MuSax Compiled Binary (`.MSB` / `.MSX`)
We will encapsulate the compiled data block in a standardized file format containing metadata for load-time resolution.

```text
Offset | Size     | Name        | Description
-------|----------|-------------|--------------------------------------------
0x00   | 4 bytes  | MAGIC       | Identification signature: "MUS\x01"
0x04   | 2 bytes  | ORG_ADDR    | Assembled target address (e.g., #8000)
0x06   | 2 bytes  | HDR_OFFSET  | Offset from data start to the song header
0x08   | 2 bytes  | DATA_LEN    | Byte length of the Z80 compiled song data
0x0A   | 1 byte   | FILE_TYPE   | Type: #80 = SONG, #81 = FX
0x0B   | 10 bytes | SONG_NAME   | ASCII song title (padded with spaces)
0x15   | 5 bytes  | COMPOSER    | ASCII author tag (padded with spaces)
0x1A   | N bytes  | DATA        | Compiled raw binary bytecode
```

---

### Playback Execution Architectures in MSX-DOS

We propose three approaches to handle the absolute memory address issue during DOS execution:

#### Alternative A: Target Segment Mapper Allocation (DOS 2 / Nextor) — *Recommended*
This is the cleanest and most scalable approach for modern MSX-DOS environments (DOS 2 or Nextor). It leverages mapper memory segments to avoid any address conflicts.

1. **Compilation Standard**: All songs are assembled to target Page 2 (`#8000` to `#BFFF`) by using `ORG #8000`.
2. **Execution Steps**:
   - The player tool `MUSAX.COM` is loaded in the standard Transient Program Area (TPA) at `#0100` and executes there.
   - The MuSax player tool uses the DOS 2 Mapper Manager API to allocate a free 16KB segment (`ALL_SEG`).
   - The player maps this segment into Page 2 (`#8000-#BFFF`) using `PUT_PH`.
   - The player loads the compiled `.MSB` file, verifies the header, and reads the raw data block directly into RAM at `#8000`.
   - The player hooks the timer interrupt (`HTIMI` at `#FD9F` or CPU interrupt vector `#0038`) to call `MUSUPDAT` and `MUSCOMM` at 50/60Hz.
   - Playback starts via `CALL MUSINIT` (with the song pointer set to `#8000 + HDR_OFFSET`).
   - When the user presses `ESC` or `SPACE`, the player stops the sound, restores the interrupt hook, releases the mapper segment using `FRE_SEG`, and exits to MSX-DOS.
3. **Pros**: No relocation code needed; zero risk of memory collision; simple build process; allows songs to be up to 16KB.
4. **Cons**: Requires a mapper-equipped MSX (standard on MSX2/DOS2/Nextor, but unavailable on stock unexpanded MSX1 machines running DOS 1).

```text
MSX RAM Layout during DOS 2 Playback:
┌──────────────────┐ #FFFF
│ Page 3 (System)  │   -> Contains Jukebox Driver Variables & Stack
├──────────────────┤ #C000
│ Page 2 (MAPPED)  │   -> Song Data Segment (Loaded at #8000)
├──────────────────┤ #8000
│ Page 1 (TPA)     │   -> MUSAX.COM Player & Sound Engine Driver Code
├──────────────────┤ #4000
│ Page 0 (TPA/DOS) │   -> MSX-DOS System Kernel & TPA
└──────────────────┘ #0000
```

#### Alternative B: Dynamic Load-Time Relocation (Universal MSX-DOS 1/2) — *User Selected*
This approach allows the song to be loaded at any arbitrary location in memory on any MSX, including stock MSX1 machines running DOS 1.

##### The `msl2msb` Tool Pipeline
The compilation and relocation table generation will be fully automated:
1. **Compilation**: The tool `msl2msb.py` compiles the `.msl` source to Z80 bytecode.
2. **Double-Pass Assembler Check**: 
   - The tool assembles the bytecode targeting `ORG #0000` (Bin A) and `ORG #0100` (Bin B) using `sjasmplus`.
   - It performs a byte-by-byte comparison of Bin A and Bin B.
   - Any 16-bit word that differs by `#0100` between the two compilations represents an absolute address.
   - The offset of each identified pointer is stored.
3. **Packaging**: The tool writes the `.MSB` file with the custom header, followed by the raw Bin A data block, followed by the relocation table (a list of 16-bit offsets terminated by `#FFFF`).

##### The Z80 Relocation API (`MSBPREP`)
To abstract the relocation logic from the DOS player loader, we will expose a light Z80 driver routine: `MSBPREP`.

```assembly
; - MSBPREP -----------------------------------------------------------------
; - COPIES AND RELOCATES A LOADED .MSB FILE FROM RAM BUFFER TO EXECUTION AREA
; - IN:  HL = POINTER TO LOADED .MSB DATA IN RAM BUFFER
; -      DE = TARGET DESTINATION FOR RUNTIME EXECUTION (MSBDEST)
; - OUT: HL = ABSOLUTE POINTER TO THE RELOCATED SONG HEADER (READY FOR MUSPLAY)
; -      CARRY = 1 IF INVALID MAGIC SIGNATURE OR METADATA CORRUPTION
; -      CARRY = 0 IF SUCCESSFUL
; - CLOBBERS: AF, BC, DE
; - -------------------------------------------------------------------------
```

###### Execution Flow & Integration Example
```assembly
; --- LOAD AND PREPARE SONG
                LD      HL,MSBLOADED    ; Pointer to raw file data in load buffer
                LD      DE,MSBDEST      ; Execution destination address
                CALL    MSBPREP         ; Relocate and get Header
                JR      C,.LOADERR      ; Jump if Carry flag is set (error)

; --- HL NOW HOLDS THE RELOCATED HEADER PTR (e.g., MSBDEST + HDR_OFFSET)
                LD      A,1             ; Play once (or 0 for loop)
                CALL    MUSPLAY         ; Initialize playback
                RET
```

###### Z80 Reference Implementation Plan for `MSBPREP`
```assembly
MSBPREP:        ; 1. Check Magic Signature
                LD      A,(HL)
                CP      "M"
                JR      NZ,.ERR
                INC     HL
                LD      A,(HL)
                CP      "U"
                JR      NZ,.ERR
                INC     HL
                LD      A,(HL)
                CP      "S"
                JR      NZ,.ERR
                INC     HL
                LD      A,(HL)
                CP      1               ; Version 1
                JR      NZ,.ERR
                INC     HL              ; HL now points to ORG_ADDR (offset 4)

                ; 2. Read Header Metadata
                LD      C,(HL)
                INC     HL
                LD      B,(HL)          ; BC = ORG_ADDR (Original assembly address)
                INC     HL
                
                LD      E,(HL)
                INC     HL
                LD      D,(HL)          ; DE = HDR_OFFSET
                PUSH    DE              ; Save HDR_OFFSET on stack
                INC     HL
                
                LD      E,(HL)
                INC     HL
                LD      D,(HL)          ; DE = DATA_LEN
                INC     HL              ; Skip FILE_TYPE (offset 10)
                
                ; Skip metadata fields (15 bytes: Name (10) + Composer (5))
                ; HL starts at offset 11. 11 + 15 = 26 (start of binary data)
                PUSH    BC              ; Save ORG_ADDR
                LD      BC,15
                ADD     HL,BC
                POP     BC              ; Restore ORG_ADDR -> BC
                
                ; 3. Setup Block Copy
                ; At this point:
                ;   HL = Pointer to raw data block (MSBLOADED + 26)
                ;   DE = DATA_LEN
                ;   BC = ORG_ADDR
                ;   Stack top = HDR_OFFSET
                ; We need to copy raw data to target destination (MSBDEST)
                ; Let's retrieve target MSBDEST parameter from caller (passed in DE originally)
                ; Since DE was clobbered by DATA_LEN, we should preserve it at entry.
```
*(A detailed register-safe Z80 flow will be added to the driver integration codebase during the harness phase, ensuring HL correctly ends pointing to `MSBDEST + HDR_OFFSET` and registers are preserved safely).*

4. **Pros**: Works on any MSX machine (even an unexpanded MSX1 with 64KB RAM and DOS 1).
5. **Cons**: Requires the double-pass compilation check tool on the PC side; loading has a tiny CPU overhead to patch the pointers (negligible, ~few thousand T-states before playback starts).


#### Alternative C: Fixed Address Region Allocation (Simplest DOS 1 Approach)
A simpler version of Alternative A that works on DOS 1 without a mapper, by reserving a fixed RAM window for the song data.

1. **Compilation Standard**: All songs are compiled to target a fixed high RAM address, e.g., `#C000` (Page 3).
2. **Execution Steps**:
   - The driver and `MUSAX.COM` reside in Page 1 / Page 2 (`#0100-#7FFF`).
   - The player loads the song bytecode directly to `#C000` (which must be verified as unused by the system) and plays it from there.
3. **Pros**: Extremely simple, no mapper or relocation required.
4. **Cons**: Highly fragile. Page 3 contains system variables, MSX stack, and DOS disk buffers. The available space is extremely restricted (usually less than 4-8KB depending on the system configuration), causing crash risks if a song is too large.

### Standalone Player Command Line Interface
The proposed MSX-DOS utility `MUSAX.COM` will have the following CLI structure:

```bash
MUSAX.COM <FILENAME.MSB> [/S] [/F]
```

- `FILENAME.MSB`: The compiled song file.
- `/S`: Silent mode (disable layout rendering, play in background while showing a simple status bar).
- `/F`: Fade-out on exit (applies `MUSFADE` when a key is pressed to exit gracefully instead of stopping abruptly).

During interactive playback, the screen will display:
- Song Title and Composer metadata.
- Real-time PSG register activity or a simple 3-channel volume meter.
- Elapsed time and current BPM.
- Keys to pause (`P`), resume (`R`), or stop/exit (`ESC` / `SPACE`).

## 4. Optional Driver Module: `MSBLOAD.Z8A`

To keep the core sound driver `MUSAX.Z8A` as lightweight as possible for ROM-only games (where size is critical and songs are embedded as `.Z8A` natively), the `.MSB` relocation capabilities will be isolated into an optional driver module: `MSBLOAD.Z8A`.

Developers who want to load songs dynamically from disk (e.g., in MSX-DOS tools or large disk-based games) can simply include this module alongside the driver.

### Module Structure & Source Code Layout
The file `src/driver/MSBLOAD.Z8A` will contain the `MSBPREP` subroutine. It will not depend on any extra RAM variables, performing all its calculations using Z80 registers (`IX`, `IY`, `HL`, `DE`, `BC`) and the CPU stack to ensure it is thread-safe and reentrant.

### Detailed Z80 Algorithm Design for `MSBPREP`
Below is the planned register allocation and assembly flow for the `MSBLOAD.Z8A` module:

```assembly
; ===========================================================================
; MUSAX DRIVER - OPTIONAL MSB LOADER LAYER
; ===========================================================================

; - MSBPREP -----------------------------------------------------------------
; - RELOCATES A LOADED .MSB FILE IN-PLACE WITHIN ITS LOAD BUFFER IN RAM
; - IN:  HL = POINTER TO LOADED .MSB DATA IN RAM (MSBSONG)
; - OUT: HL = ABSOLUTE POINTER TO THE RELOCATED SONG HEADER (READY FOR MUSPLAY)
; -      CARRY = 1 IF INVALID MAGIC SIGNATURE OR METADATA CORRUPTION
; -      CARRY = 0 IF SUCCESSFUL
; - CLOBBERS: AF, BC, DE
; - -------------------------------------------------------------------------
MSBPREP:        PUSH    IX
                PUSH    IY
                PUSH    HL              ; Save original MSBSONG pointer to stack

                ; 1. Check Magic Signature via Loop (Saves 7 bytes of code)
                LD      DE,.SIGN
                LD      B,4
.SIGNLP:        LD      A,(DE)
                CP      (HL)
                JR      NZ,.ERR_POP     ; Pop original pointer and return error
                INC     HL
                INC     DE
                DJNZ    .SIGNLP

                ; HL now points to ORG_ADDR (offset 4)

                ; 2. Read Metadata from Header
                LD      C,(HL)
                INC     HL
                LD      B,(HL)          ; BC = ORG_ADDR (original assembly target)
                INC     HL
                
                LD      E,(HL)
                INC     HL
                LD      D,(HL)          ; DE = HDR_OFFSET
                PUSH    DE              ; Stack = [MSBSONG, HDR_OFFSET]
                INC     HL
                
                LD      E,(HL)
                INC     HL
                LD      D,(HL)          ; DE = DATA_LEN
                PUSH    DE              ; Stack = [MSBSONG, HDR_OFFSET, DATA_LEN]
                
                ; 3. Setup Pointers
                ; Recover original MSBSONG pointer from stack to compute MSBDEST
                ; Stack is currently: [MSBSONG, HDR_OFFSET, DATA_LEN]
                LD      HL,4
                ADD     HL,SP           ; HL points to MSBSONG in the stack (offset 4)
                LD      E,(HL)
                INC     HL
                LD      D,(HL)          ; DE = MSBSONG (original buffer start)
                
                ; Calculate MSBDEST (execution destination) = MSBSONG + 26
                LD      HL,26
                ADD     HL,DE           ; HL = MSBDEST (bytecode starts after 26b header)
                PUSH    HL
                POP     IX              ; IX = MSBDEST
                
                ; Compute DELTA = MSBDEST - ORG_ADDR
                ; HL has MSBDEST. BC has ORG_ADDR.
                OR      A
                SBC     HL,BC           ; HL = DELTA (offset shift)
                PUSH    HL
                POP     IY              ; IY = DELTA
                
                ; Compute Relocation Table Address (TABLE_ADDR = MSBDEST + DATA_LEN)
                ; Pop DATA_LEN into BC (Stack becomes [MSBSONG, HDR_OFFSET])
                POP     BC              ; BC = DATA_LEN
                PUSH    IX
                POP     HL              ; HL = MSBDEST
                ADD     HL,BC           ; HL = TABLE_ADDR (start of Relocation Table)
                
                ; Now we have:
                ;   HL = Pointer to read from Relocation Table
                ;   IX = MSBDEST
                ;   IY = DELTA
                ;   Stack = [MSBSONG, HDR_OFFSET]

                ; 4. Process Relocation Table
.LOOP:          LD      C,(HL)
                INC     HL
                LD      B,(HL)
                INC     HL              ; BC = Offset of pointer to relocate
                
                LD      A,B
                AND     C
                CP      #FF             ; Is offset #FFFF (end of table)?
                JR      Z,.DONE
                
                ; Relocate pointer: PTR_ADDR = MSBDEST + BC
                PUSH    HL              ; Save table pointer
                PUSH    BC              ; Save offset
                
                PUSH    IX
                POP     HL              ; HL = MSBDEST
                ADD     HL,BC           ; HL = PTR_ADDR
                
                ; Read the absolute 16-bit address at (PTR_ADDR)
                LD      C,(HL)
                INC     HL
                LD      B,(HL)          ; BC = Old pointer value
                DEC     HL              ; HL back to PTR_ADDR (low byte)
                
                ; Add DELTA (in IY) to old pointer (BC)
                PUSH    IY
                POP     DE              ; DE = DELTA
                PUSH    HL              ; Save PTR_ADDR
                
                EX      DE,HL           ; HL = DELTA, DE = PTR_ADDR
                ADD     HL,BC           ; HL = DELTA + Old pointer = New pointer
                EX      DE,HL           ; DE = New pointer, HL = PTR_ADDR
                
                ; Write relocated pointer back to (PTR_ADDR)
                LD      (HL),E
                INC     HL
                LD      (HL),D
                
                POP     HL              ; Clean PTR_ADDR from stack
                POP     BC              ; Restore offset
                POP     HL              ; Restore table pointer
                JR      .LOOP

.DONE:          ; 5. Success: Compute final Song Header address (HL = MSBDEST + HDR_OFFSET)
                ; Stack currently holds [MSBSONG, HDR_OFFSET]
                POP     DE              ; DE = HDR_OFFSET
                POP     BC              ; BC = MSBSONG (discarded)
                
                PUSH    IX
                POP     HL              ; HL = MSBDEST
                ADD     HL,DE           ; HL = MSBDEST + HDR_OFFSET
                
                POP     IY
                POP     IX
                OR      A               ; Clear carry flag (success)
                RET

.ERR_POP:       POP     HL              ; Discard original MSBSONG from stack
.ERR:           POP     IY
                POP     IX
                SCF                     ; Set carry flag (error)
                RET

.SIGN:          DEFB    "MUS", 1
```

## 5. MSX-DOS Player Command Line Utility (`MUSAX.COM`)

This section outlines the detailed requirements, CLI parameter parsing, non-blocking keyboard loop, console interface design, and interrupt-based state tracking for the standalone player.

### Command Line Invocation Syntax
```bash
MUSAX.COM <SONG.MSB> [<FXS_BANK.MSB>] [options]
```

#### Supported Arguments:
1. `<SONG.MSB>`: Mandatory path to the compiled background music (BGM) module (`TYPE_SONG = #80`).
2. `[<FXS_BANK.MSB>]`: Optional second parameter. Path to the compiled sound effects (SFX) bank module (`TYPE_FX = #81`).
   - If provided, `MUSAX.COM` will load and relocate this file *in-place* in a separate memory buffer.
   - It will update the MuSax driver pointer variable `(FXPTR)` with the address of the relocated FX table inside the module.
   - This allows testing real-time sound effect priorities and ghost mixing concurrently with BGM.
3. `/L[:count]`: Loop modifier.
   - If `/L` is omitted, the song plays **once** and exits to DOS automatically.
   - If `/L` is passed without parameters, the song loops **infinitely** (loops = 0 passed to `MUSPLAY`).
   - If `/L:N` is passed (e.g. `/L:3`), the song loops exactly **N times** before exiting.
4. `/F`: Graceful exit. If the user stops playback manually (pressing `ESC` or `SPACE`), the utility performs a quick fade-out using the `MUSFADE` API instead of cutting the PSG sound abruptly.
5. `/S`: Silent Mode. Background execution. The utility silences console output completely. This allows calling `MUSAX.COM` inside `.BAT` scripts to play background music before starting another game or utility.
6. `/H` or `/?`: Shows help information including the syntax, options, and keyboard commands, then exits immediately.

### Interactive Sound FX Triggering & Paging
When an `<FXS_BANK.MSB>` is loaded, the user can trigger sound effects in real-time. Since a sound bank can contain more than 10 effects (e.g. 11, 20, or more), we implement a **paging system** using the Cursor keys:

- **Keys `1` to `9` and `0`**: Trigger SFX from the active page.
  - Page 0: triggers SFX IDs `0` to `9`.
  - Page 1: triggers SFX IDs `10` to `19`.
  - Page 2: triggers SFX IDs `20` to `29`, etc.
- **Row 8, Bit 6 (`LEFT`) and Bit 7 (`RIGHT`)**: Changes the active SFX page.
  - Pressing `RIGHT` increments the active page (if there are more FXs in the bank).
  - Pressing `LEFT` decrements the active page (minimum page 0).
- **Trigger Logic**:
  - Let `PAGE` be the active page (starts at 0).
  - When a number key `N` is pressed, let `ID = N - 1` (with `0` key mapping to index 9).
  - Calculate `SFX_ID = (PAGE * 10) + ID`.
  - If `SFX_ID` is within the maximum number of sound effects in the bank, set `A = SFX_ID` and execute `CALL FXPLAY`.

#### Element Dimensioning & Display Rules:
- **Title, Author**: Extracted directly from the `.MSB` header metadata.
- **Time**: Calculated by tracking frame ticks in the interrupt handler.
- **BPM, Loop, Fade**: Read directly from the driver variables `GLBBPM`, `GLBLOOP`, `GLBFADSP`.
- **Channel Meters**: A simplified 15-character text progress bar mapping the volume of the three music channels (A, B, C) using solid `#DB` (`█`) and shaded `#B1` (`░`) ASCII blocks, followed by the decimal volume value and the active note string. Read directly from the channel states.

#### High-Performance SCREEN 0 Rendering Architecture
Updating a 40x24 real-time text UI at 50/60Hz using standard BDOS calls (such as Function `09h` string printing or character-by-character printing) is extremely slow. The operating system overhead (handling scrolling, boundary checking, and system checks) will choke the Z80 CPU, causing stuttering in both the music and the visuals.

To bypass this bottleneck, `MUSAX.COM` will adopt a double-buffered VRAM writing strategy:
1. **RAM Framebuffer (`DOSBUFF`)**:
   - The utility will maintain a 960-byte RAM buffer (40 columns × 24 rows) representing the SCREEN 0 screen layout.
   - All string formatting, volume meters, and elapsed time text will be written directly to `DOSBUFF` in CPU RAM using register pointers (fast assembly operations).
2. **Dynamic VRAM Target Resolution**:
   - Instead of hardcoding the SCREEN 0 VRAM address (which can change depending on DOS page configuration or width modifications), the utility will read the system variable `TXNAMB` (address `#F3B3` in the BIOS RAM variables area).
   - `TXNAMB` contains the 16-bit base address of the SCREEN 0 Name Table in VRAM (typically `#0000` or `#F000`).
3. **Block Dump to VRAM**:
   - At the end of each drawing cycle, if the layout has changed (checked via a `DOSDIRTY` flag), the utility will execute a block copy from `DOSBUFF` to the VRAM destination `(TXNAMB)` using an optimized block write loop (e.g. using `LDIRVM` or unrolled `OUTI` blocks writing to VDP ports resolved from BIOS variables `#0006` and `#0007`).
   - Transferring 960 bytes directly to VRAM takes less than 20,000 T-states, consuming less than **3% of the total CPU frame time** on a standard 3.58MHz MSX, resulting in smooth, flicker-free rendering.
4. **DOS Restoration on Exit**:
   - When exiting playback, the player utility will clear the screen using standard BIOS calls (`CALL CLRSPR` / `INITXT`) and restore the cursor position before returning control to the command line, ensuring the DOS environment remains clean and stable.

### DOS Non-Blocking Keyboard Loop
Since MSX-DOS file input and execution are block-based, we must scan the keyboard in a non-blocking loop during playback. We will use standard BDOS function calls to maintain full operating system compatibility:

1. **Check Input Status (BDOS Function `0Bh`)**:
   - Checks if a key has been pressed and is waiting in the input buffer.
   - Output: `A = #FF` (Key waiting) or `A = #00` (No key).
2. **Read Buffer (BDOS Function `07h`)**:
   - Read the character from the buffer without printing it to the console (direct console input without echo).
   - Output: `A = character ASCII code`.

#### Keyboard Control Processing Flow:
- **`SPACE` or `P`**: Toggle Pause.
  - Toggles the pause bit in `MUSSTAT` (Bit 3).
  - When paused, the mixer commits 0 volume to all PSG registers.
  - Updates `Status` field to `PAUSED`.
- **`ESC` or `S`**: Stop & Exit.
  - If `/F` (Fade Out option) is active: trigger `MUSFADE(0, 5)` to fade out volume, wait until fade reaches 0, then exit.
  - If `/F` is inactive: call `MUSSTOP` immediately.
  - Restore timer interrupt hook and exit to DOS.
- **`F`**: Trigger manual Fade Out.
  - Call `MUSFADE(0, 3)`. The driver will automatically call `MUSSTOP` and trigger an exit when the fade completes.
- **`I`**: Trigger manual Fade In.
  - Snap master volume to 0, start playback, and call `MUSFADE(255, 3)` to fade in.

### Frame-Time and Loop Tracking
To display the time elapsed (`Time`) and track loops:
1. **Interrupt Vector Hook (`HTIMI` at `#FD9F` or CPU interrupt vector `#0038`)**:
   - The handler runs at 50Hz (PAL) or 60Hz (NTSC).
   - In each tick, we increment a 2-byte frame counter in RAM (`FRAMETICK`).
2. **Frecuency Detection**:
   - Read the MSX system variable `RG0SAV` (VDP register 0 save) or BIOS configurations at boot to detect if the VDP is running in PAL (50Hz) or NTSC (60Hz) mode.
   - Track ticks per second: `TICKS_PER_SEC = 50` or `60`.
3. **Time Calculation**:
   - In the main drawing loop:
     `Seconds = FRAMETICK / TICKS_PER_SEC`
     `Minutes = Seconds / 60`
     `Seconds = Seconds % 60`
   - Convert to decimal characters and display in the `MM:SS` format.
4. **Auto-Exit Condition**:
   - If not looping infinitely, compare the driver's current loop index against the limit.
   - When the song finishes (`MUSSTAT` bit 0 becomes 0, indicating execution stopped), trigger the exit routine.
